# Tag Data Migration (Blog + Product) — Design

**Goal:** Migrate WordPress/WooCommerce tags into Postgres for both blog articles
and products, so tags exist as first-class data. This is the **data-only**
slice: export → source record → schema → idempotent importer, for two parallel
entities. Public surfaces and admin management are out of scope (later specs).

## Background

- The source WordPress/Woo DB models everything as terms partitioned by taxonomy
  (`category`/`post_tag` for posts, `product_cat`/`product_tag` for products).
  Tags are just terms under the `post_tag` / `product_tag` taxonomies.
- Our app deliberately does **not** mirror WP's unified terms — it keeps a
  dedicated `blog_category` table and a separate product `category` table. Tags
  follow the same separation: **per-entity tag tables**, decided with the user.
- Tags are **not exported or modeled today**: `export-blog.py` and the catalog
  `export.py` query only `category`/`product_cat`; `SourceBlogPost` carries only
  `categorySlugs`, `SourceProduct` only `categoryWooTermIds`. So each migration
  is a full vertical slice.

## Scope

In scope — two independent data migrations, each mirroring that entity's
existing **category** handling:

- **Blog tags** keyed by **slug** (mirrors `blog_category`).
- **Product tags** keyed by **Woo term id** (mirrors product `category`).

Out of scope (named, deferred to later specs):

- Public surfaces: blog tag pages (`/blog/cimke/{slug}`), product tag
  display/filtering, and the sidebar tag cloud deferred in the sidebar work.
- Admin tag management (rename/merge/hide) in the Refine SPA.

## Sequencing decision

The user chose "both migrations first": this spec covers blog-tag AND
product-tag data migration together (one coherent phase: get tags into
Postgres). Surfaces and admin come as separate specs afterward.

## Architecture

### A. Blog tags (keyed by slug — mirrors `blog_category`)

**Export** — `scripts/woo-export/export-blog.py`:
- Add a `taxonomy = 'post_tag'` terms query producing a `tags` list of
  `{name, slug}` (parallel to the existing `category` query at lines ~76-80).
- Add a `post_tag` term_relationships query producing `tags_by_post` (parallel
  to `cats_by_post`), and emit `"tagSlugs": tags_by_post.get(p["id"], [])` on
  each post (parallel to `categorySlugs`).
- Top-level JSON gains a `"tags"` key alongside `"posts"` and `"categories"`.

**Source records** (`integrations/wordpress`):
- New `record SourceBlogTag(String name, String slug)` (parallel to
  `SourceBlogCategory`).
- `SourceBlog` gains `List<SourceBlogTag> tags`.
- `SourceBlogPost` gains `List<String> tagSlugs` (parallel to `categorySlugs`).

**Schema** — Flyway **V25** (`V25__blog_tags.sql`):
```sql
CREATE TABLE blog_tag (
    id   BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name TEXT NOT NULL,
    slug TEXT NOT NULL UNIQUE
);
CREATE TABLE blog_post_tag (
    blog_post_id BIGINT NOT NULL REFERENCES blog_post (id) ON DELETE CASCADE,
    blog_tag_id  BIGINT NOT NULL REFERENCES blog_tag (id)  ON DELETE CASCADE,
    PRIMARY KEY (blog_post_id, blog_tag_id)
);
```
(Parallel to `blog_category` / `blog_post_category` in V22.)

**Domain** (`domain/blog`):
- `BlogTag` entity (`id`, `name`, `slug`) + `create(name, slug)` factory
  validating the slug via `BlogPost.isValidSlug` — parallel to `BlogCategory`.
- `BlogTagRepository extends JpaRepository` with `findBySlug`, `existsBySlug`.
- `BlogPost` gains a `Set<BlogTag> tags` `@ManyToMany` (join `blog_post_tag`),
  mirroring how `categories` is mapped.

**Importer** — `BlogImporter`:
- `upsertTags(SourceBlog, report)` → `Map<String, BlogTag>` by slug, mirroring
  `upsertCategories` (findBySlug → create if absent; existing kept).
- In `upsertPost`, resolve `source.tagSlugs()` against that map and set the
  post's tags (mirroring the existing category resolution). Idempotent (rule
  #4): re-running upserts without duplicates; a re-import re-links the same
  tags.

### B. Product tags (keyed by Woo term id — mirrors product `category`)

**Export** — `scripts/woo-export/export.py`:
- Add a `taxonomy = 'product_tag'` terms query producing a `tags` list of
  `{wooTermId, slug, name}` (parallel to the `product_cat` query at lines
  ~75-88), and a `product_tag` term_relationships query giving per-product tag
  term ids (parallel to how `categoryWooTermIds` is assembled).
- Top-level JSON `SourceCatalog` gains a `"tags"` array; each product gains
  `"tagWooTermIds"`.

**Source records** (`integrations/woo`):
- New `record SourceTag(long wooTermId, String slug, String name)` (a lean
  parallel to `SourceCategory`; tags are flat — no parent/sortOrder/SEO).
- `SourceCatalog` gains `List<SourceTag> tags`.
- `SourceProduct` gains `List<Long> tagWooTermIds` (parallel to
  `categoryWooTermIds`).

**Schema** — Flyway **V26** (`V26__product_tags.sql`):
```sql
CREATE TABLE product_tag (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    external_id BIGINT NOT NULL UNIQUE,   -- Woo term id
    slug        TEXT NOT NULL,
    name        TEXT NOT NULL
);
CREATE TABLE product_tag_map (
    product_id     BIGINT NOT NULL REFERENCES product (id)     ON DELETE CASCADE,
    product_tag_id BIGINT NOT NULL REFERENCES product_tag (id) ON DELETE CASCADE,
    PRIMARY KEY (product_id, product_tag_id)
);
```
(`product_tag` = tag entity, mirroring `category`; `product_tag_map` = join,
parallel to `product_category`. Named `_map` to avoid a `product_product_tag`
collision with the entity table name.)

**Domain** (`domain/catalog`):
- `ProductTag` entity (`id`, `externalId`, `slug`, `name`) + `create(externalId,
  slug, name)` — parallel to `Category`.
- `ProductTagRepository` with `findByExternalId`.
- `Product` gains a `Set<ProductTag> tags` `@ManyToMany` (join
  `product_tag_map`), mirroring `categories`.

**Importer** — `CatalogImporter`:
- `upsertTags(SourceCatalog, report)` → `Map<Long, ProductTag>` by
  `externalId`, mirroring `upsertCategories` (findByExternalId → create/update).
- In `upsertProduct`, resolve `source.tagWooTermIds()` against that map and set
  the product's tags (mirroring the category linking via
  `categoryByExternalId`). Idempotent by Woo term id / product id.

## Migration versioning (collision lesson)

Tags take the next contiguous versions on `main`: **V25** (`blog_tags`), **V26**
(`product_tags`). `main`'s current latest is V24 (`sidebar_blocks`).

The unmerged `instagram-feed` worktree currently holds an on-disk `V25__instagram_feed.sql`
that is **not applied to any database** (its dev-DB rows/tables were removed
during the sidebar work). When `instagram-feed` merges, renumber it to the next
free version at that time — versions follow **merge order**, and the branch that
merges last takes the higher number. This avoids the checksum/out-of-order
collision that blocked startup earlier.

## Import reporting

Extend the existing import reports with tag counters, mirroring category
counters:
- `BlogImportReport`: `tagCreated()` (+ a count field), analogous to
  `categoryCreated()`.
- Catalog `ImportReport`: `tagCreated()` / `tagUpdated()`, analogous to
  `categoryCreated()`/`categoryUpdated()`.

## Testing

- **Exporter unit tests** (Python, mirroring `test_export_blog.py`):
  - `export-blog.py`: a post with a `post_tag` term yields the post's
    `tagSlugs` and the top-level `tags` list; the JSON keys include `"tags"`.
  - `export.py`: a product with a `product_tag` term yields `tagWooTermIds` and
    the top-level `tags` list.
- **Importer ITs** (Testcontainers Postgres, mirroring `BlogImporterIT` /
  `CatalogImporter` ITs):
  - Blog: importing a `SourceBlog` with tags creates `blog_tag` rows and links
    them to the post; re-import is idempotent (no duplicate tags, same links);
    an unknown tag slug on a post is created; a post with no tags links none.
  - Product: importing a `SourceCatalog` with tags creates `product_tag` rows
    keyed by `external_id` and links products; re-import idempotent; a tag
    rename in the source updates the existing row (findByExternalId), not a
    duplicate.
- No public-surface or admin tests (out of scope).

## Constraints honored

- **Idempotency (rule #4):** both importers upsert tags without duplication on
  re-run (blog by slug, product by Woo term id / SKU-product).
- **Slugs stable (rule #7):** tag slugs are taken 1:1 from the source.
- **No public-render change:** this slice only adds data; `SidebarQueryService`,
  the blog fragment, and product pages are untouched.
- **Enums/columns, Testcontainers, `-Dskip.frontend=true`** per existing
  backend conventions. Code/comments/commits English.
