# Social Links as a Site-Wide Setting — Design

**Goal:** Promote social links from the blog-sidebar SOCIAL block to a site-wide
setting: one source of truth, edited under `/admin/settings` (a dedicated
`/settings/social` screen), displayed in the public site top bar (left of the
address line) and still in the blog sidebar SOCIAL block.

## Background

- Social links currently live in the SOCIAL `sidebar_block` content JSON
  (`SocialContent(title, links[{network,url}])`), edited in `/admin/sidebar-blocks`
  and rendered in the blog sidebar. The instagram merge generalized the icon
  rendering to "any network" with one-line admin rows.
- `/admin/settings` is a **static prototype** (`data/settings.ts`: "Static
  example data. No backend wiring") — tabs render hardcoded values, no REST, no
  persistence. So the social editor added here is **new functional** code, not a
  slot in the mock tabs.
- The public header fragment (`templates/fragments/layout.html :: header`) has a
  `.topbar > .wrap` with a left region (address/hours line) and a right region
  (free-shipping note).

## Decisions (confirmed with the user)

1. Display icons in the **public site top bar**, left region.
2. **Dedicated site-setting store** for social links (single source of truth);
   the SOCIAL sidebar block stops owning its own `links` JSON and reads the store.
3. **Keep** the blog sidebar SOCIAL block, reading the shared links.
4. Admin editor lives at a **separate `/settings/social` route** (not a tab in
   the static Settings sub-nav).
5. In the topbar, **keep the address line**; add the icons to its **left**.

## Architecture

### Data model (Flyway V30)

`social_link` — the single source of truth:
```sql
CREATE TABLE social_link (
    id            BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    network       TEXT NOT NULL,   -- facebook | instagram | youtube | pinterest | ...
    url           TEXT NOT NULL,
    display_order INT  NOT NULL
);
```
**Seeded** from the current SOCIAL block links so nothing is lost:
`facebook → https://www.facebook.com/Nemiskacat` (order 1),
`instagram → https://www.instagram.com/nemiskacat/` (2),
`youtube → https://www.youtube.com/user/Nemiskacatponthu` (3).

Domain: `SocialLink` entity (`id, network, url, displayOrder`) + setters for
`url`/`network`/`displayOrder`; `SocialLinkRepository.findAllByOrderByDisplayOrderAsc()`.

Migration is **V30** (main is at V29 after the tag merge). Same cross-branch
caveat: if another in-flight branch grabs V30, versions follow merge order.

### Read side

`SocialLinkQueryService` (`@Service @Transactional(readOnly=true)`):
- record `SocialLinkView(String network, String url)`
- `List<SocialLinkView> links()` → ordered list from the repo.

Consumers:
- **Public header** — a `@ControllerAdvice(annotations = Controller.class)` (or
  scoped to the `web` package) with `@ModelAttribute("socialLinks")` returning
  `socialLinks.links()`, so the `header` fragment sees it on every public page.
  Global, no per-user data → blog/product/home HTML stays cacheable (CLAUDE.md
  #2). *(Confirm there is no existing global `@ModelAttribute` advice to extend;
  otherwise add one narrowly scoped to web controllers.)*
- **Blog sidebar SOCIAL block** — `SidebarQueryService` builds the SOCIAL
  `SocialContent(title, links)` with `title` from the block JSON and `links`
  from `SocialLinkQueryService` (mirrors how CATEGORIES attaches its live list).
  The block JSON keeps only `title`; its stored `links` are ignored.

### Public header (Thymeleaf)

- Extract the per-network inline-SVG social-icon rendering into a **shared
  fragment** `fragments/components :: socialIcons(links)` (DRY — reused by the
  header and the blog sidebar SOCIAL section, which currently inlines the SVGs).
- In `layout.html :: header`, render `socialIcons(${socialLinks})` in the
  `.topbar` left region, **before** the existing address/hours line (icons left,
  address to their right). CSS in `site.css` for the topbar icon strip.
- The blog sidebar SOCIAL section (`fragments/blog.html`) switches to the shared
  `socialIcons` fragment fed by `block.social.links`.

### Admin editing (`/admin/settings/social`)

Backend — new REST resource mirroring the sidebar-blocks controller conventions
(path-based ADMIN auth, plain-string errors via `AdminExceptionHandler`,
`X-Total-Count`, audit every write):
- `SocialLinkAdminService` (`@Transactional`): `list()`, `create(SocialLinkUpsert)`,
  `update(id, SocialLinkUpsert)`, `delete(id)`, `reorder(List<Long> ids)`;
  records `SocialLinkView(id, network, url, displayOrder)`, `SocialLinkUpsert(network, url)`,
  `ReorderRequest(List<Long> ids)`; a `NotFoundException`.
- `SocialLinkController` under `/api/admin/settings/social`:
  `GET` (+X-Total-Count), `GET /{id}`, `POST` (create), `PUT /{id}`,
  `DELETE /{id}`, `POST /reorder`.
- `AdminExceptionHandler` gains a `SocialLinkAdminService.NotFoundException → 404`.

admin-ui — a **new functional screen** at route `/settings/social`:
- Refine resource `settings/social` (or `social-links`) → `/api/admin/settings/social`.
- Screen: a table of `{network, url}` rows with add / remove / edit / ↑↓ reorder
  (reuse the sidebar-blocks list + apiFetch reorder pattern). Refine
  `useTable`/`useCreate`/`useUpdate`/`useDelete`; `apiFetch` for reorder.
- Route + resource registered in `App.tsx`; a nav entry (near Settings) linking
  to `/settings/social`; i18n namespace (hu/en).
- The **SOCIAL sidebar-block editor** (`pages/sidebar-blocks/edit.tsx`) drops its
  links `Form.List` — the SOCIAL case now edits only the `title`. Links are
  managed in `/settings/social`.

## Testing

- **Backend:** `SocialLinkRepositoryIT` (Testcontainers) — seed present, ordered.
  `SocialLinkAdminServiceIT` — create/update/delete/reorder + NotFound.
  `SocialLinkControllerIT` — 401 unauth, list+X-Total-Count, create/update/delete,
  reorder, CSRF on writes. `SidebarQueryServiceIT` — SOCIAL block now attaches the
  store's links (not the block JSON). A web render test (MockMvc) asserting the
  public header includes the social icons / links.
- **admin-ui:** build green; a pure helper test if any content parsing is added;
  screens gated by build (project convention).

## Constraints honored

- Cacheable public HTML (rule #2): social links are global, no per-user data.
- Business logic in services (rule #1); admin auth path-based; plain-string
  admin errors; audit on writes; idempotent reorder (exact-id-set).
- Slugs/URLs 1:1; code/comments/commits English; UI strings Hungarian.
- No new framework; migration additive (V30).

## Scope / sequencing

One spec, one plan. Natural task order: (1) `social_link` schema + entity + repo
+ seed; (2) `SocialLinkQueryService` + wire SOCIAL sidebar block to it; (3)
public header — shared `socialIcons` fragment + global model attribute + CSS; (4)
`SocialLinkAdminService` + `SocialLinkController` + IT; (5) admin-ui
`/settings/social` screen + nav/i18n + drop links from the SOCIAL block editor.

## Self-review notes

- Placeholders: schema, records, endpoints, screen behavior concrete; the one
  "confirm no existing global @ModelAttribute advice" is a verification step for
  the implementer, not a design gap.
- Consistency: `SocialLink(network,url,displayOrder)` used across store →
  query service → header → sidebar → admin; the SOCIAL block JSON reduced to
  `{title}` with links sourced from the store in both render paths (header +
  sidebar).
- Scope: single feature; migration V30; public + admin + web all covered.
- Ambiguity resolved: separate `/settings/social` route; address line kept with
  icons to its left; SOCIAL sidebar block kept, reading the shared store.
